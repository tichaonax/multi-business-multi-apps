# MBM-174: Expense Account Transfers

**Date:** 2026-04-10
**Status:** IN PROGRESS
**Priority:** High

---

## Objective

Allow users with the new `canTransferBetweenAccounts` permission to transfer funds directly from one expense account to another expense account they have access to, with a full audit trail and mandatory notes.

---

## Background / Context

The codebase already has an internal concept of account-to-account fund movement. The existing "auto-transfer" mechanism (used by the supplier-payment cross-account flow in `/api/supplier-payments/requests/[id]/pay/route.ts`) creates:
- An `ExpenseAccountPayments` row with `paymentType = 'TRANSFER_OUT'` and `payeeType = 'ACCOUNT_TRANSFER'` on the source account.
- An `ExpenseAccountDeposits` row with `sourceType = 'ACCOUNT_TRANSFER'` on the destination account.

The new manual transfer feature follows the exact same two-row atomic pattern but adds:
1. A user-initiated flow (form + modal).
2. A new permission gate (`canTransferBetweenAccounts`).
3. Mandatory transfer notes.
4. Access-scoped destination account discovery (user can only see accounts they have write access to).
5. A dedicated API endpoint.

The existing transaction history display in `transactions/route.ts` already detects and labels `TRANSFER_OUT` payments and `ACCOUNT_TRANSFER` deposits correctly, so the new transfers will render correctly in the transaction history without any changes to that route.

---

## Data Model

No new Prisma model or migration is required. The feature reuses existing tables with existing discriminator columns.

**Two rows are created atomically per transfer:**

Source account — `ExpenseAccountPayments`:
```
paymentType      = 'TRANSFER_OUT'
payeeType        = 'ACCOUNT_TRANSFER'
status           = 'SUBMITTED'        -- terminal; no approval required
amount           = <transfer amount>
notes            = <user note, required>
createdBy        = <user.id>
submittedBy      = <user.id>
submittedAt      = <now>
```

Destination account — `ExpenseAccountDeposits`:
```
sourceType        = 'ACCOUNT_TRANSFER'
amount            = <transfer amount>
depositDate       = <now>
autoGeneratedNote = 'Transfer from "<source account name>" — <user note>'
createdBy         = <user.id>
```

Both rows are created inside a single `prisma.$transaction()` call, followed by `updateExpenseAccountBalanceTx()` for both accounts.

> **Note:** `BusinessTransferLedger` is NOT used for manual transfers. The ledger was designed for auto-transfers requiring settlement/return tracking. Manual transfers are immediate and symmetrical.

## Phase 3 — API Route

**New file:** `src/app/api/expense-account/[accountId]/transfer/route.ts`

Pattern: follows the atomic transfer logic in `/api/supplier-payments/requests/[id]/pay/route.ts`.

**POST body:**
```ts
{
  destinationAccountId: string  // required
  amount: number                 // required, > 0
  notes: string                  // required, non-empty
}
```

**Handler logic (all DB writes inside `prisma.$transaction`):**

1. Auth: `getServerUser()` — 401 if absent.
2. Permission: `getEffectivePermissions(user).canTransferBetweenAccounts || user.role === 'admin'` — 403 if false.
3. Source access: `canUserWriteAccount(user.id, sourceAccountId)` — 403 if false.
4. Destination access: `canUserWriteAccount(user.id, destinationAccountId)` — 403 if false.
5. Validate: source !== destination — 400.
6. Validate: `notes.trim()` non-empty — 400 "Transfer notes are required".
7. Validate: amount > 0 — 400.
8. Inside `$transaction`: authoritative balance check (aggregate deposits minus submitted/paid payments) — throw if amount > balance.
9. Create `ExpenseAccountPayments` (TRANSFER_OUT) on source.
10. Create `ExpenseAccountDeposits` (ACCOUNT_TRANSFER) on destination.
11. `updateExpenseAccountBalanceTx(tx, sourceAccountId)`.
12. `updateExpenseAccountBalanceTx(tx, destinationAccountId)`.
13. After transaction: `createAuditLog({ action: 'EXPENSE_ACCOUNT_TRANSFER', entityType: 'ExpenseAccountTransfer', ... })`.
14. Return `{ success: true, data: { sourcePaymentId, destinationDepositId, amount } }`.

**Destination account list:** The transfer modal uses the existing `GET /api/expense-account` endpoint to list accessible accounts — no new endpoint needed.

- Notes non-empty

## Phase 6 — Transfer Reports

### What to report

Two report views are needed:

**A. Account-level transfer history** — visible on the expense account detail page (new "Transfers" tab or section). Shows all transfers in and out of that specific account.

**B. Cross-account transfer report** — accessible from the sidebar Tools / profile menu. Shows all transfers the user can see across all their accessible accounts, with filtering by date range, source account, and destination account.

---

### New API Route: `GET /api/expense-account/transfers`

Query params:
- `accountId` *(optional)* — filter to transfers involving this account (source OR destination)
- `direction` *(optional)* — `'IN'` | `'OUT'` | `''` (default: both)
- `startDate` *(optional)* — ISO date
- `endDate` *(optional)* — ISO date
- `limit` *(default 50)*
- `offset` *(default 0)*

**Logic:**

Since transfers are stored as paired rows (`ExpenseAccountPayments` TRANSFER_OUT + `ExpenseAccountDeposits` ACCOUNT_TRANSFER), the report reconstructs the transfer pair by matching on:
- Same `amount`
- Same `createdBy`
- Same `createdAt` (within a 1-second window, or match on the deposit's `autoGeneratedNote` containing the source account name)

The cleanest approach is to query `ExpenseAccountPayments` where `paymentType = 'TRANSFER_OUT'` for accounts the user can access, then for each row look up the matching deposit on the destination account. This gives one unified "transfer" record per transaction pair.

Response shape per record:
```ts
{
  id: string                    // source payment id (used as the canonical transfer id)
  transferDate: string          // ISO datetime
  amount: number
  sourceAccount: { id: string; accountName: string; accountNumber: string }
  destinationAccount: { id: string; accountName: string; accountNumber: string }
  notes: string
  initiatedBy: { id: string; name: string }
}
```

Aggregates in response:
```ts
{
  totalTransferredOut: number   // sum of outbound transfers for filtered set
  totalTransferredIn: number    // sum of inbound transfers for filtered set
  count: number
}
```

Auth: `canAccessExpenseAccount || isAdmin`. Scoped to accounts the user can view — does not expose transfers for accounts outside their access.

## Out of Scope

- Approval workflow for transfers (immediate SUBMITTED status, no draft/pending state).
- Scheduling or recurring transfers.
- Reversing a transfer (separate ticket; existing reversal flow does not apply).
- `BusinessTransferLedger` rows for manual transfers.
- Mobile-specific layout changes beyond Tailwind responsive classes.
- Export to CSV/Excel for transfer reports (future).
